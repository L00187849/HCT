## Summary

Electric-Petrol.ie has expanded from a single electric service station infrastructure design into a national organisation with more than 60 service stations across Ireland. The original branch solution was designed for local operation at a time when the company had no external connectivity. Each branch used local host servers and virtualisation to provide the services required at site level.

The business now has WAN connectivity between branch sites, head office and two data centres. The customer has decided to centralise the major line of business applications at two data centres:

* **Data Center 1:** Existing co-location site in Mayo
* **Data Center 2:** Greenfield data centre in Malin, Co. Donegal
* **Head Office:** Central business and administration location
* **Branch Sites:** More than 60 service stations nationally

Although major line of business applications are being centralised, the local branch infrastructure still needs to remain resilient. The customer has specified that each branch site must continue operating in the event of a single server failure.

The existing branch design uses two Dell PowerEdge R670 servers running Windows Server 2022 Datacenter with Microsoft Hyper-V. These hosts support virtualised infrastructure services such as Active Directory, DNS, DHCP, NTP, monitoring and management services. The proposed upgrade adds a third matching Dell PowerEdge R670 server, implements clustered shared storage, strengthens failover capability and redesigns Authentication, Authorisation and Accounting (AAA) as part of the `electric-petrol.ie` domain tree.

The recommended solution is a standardised three-node Hyper-V Failover Cluster at each branch site. The cluster will use shared storage so that virtual machines are not dependent on one physical host. If one host fails, the critical virtual machines can restart or continue operating on the remaining two hosts. This design should be piloted at one branch first and then rolled out nationally in controlled phases.

## Exactly what technical tasks must be done to do the upgrade and what are the implications.

This section describes the technical work required to upgrade each branch from the existing two-host Hyper-V design to a three-node Hyper-V Failover Cluster with clustered shared storage. It also explains the operational implications of the upgrade.

### 1. Complete a branch infrastructure audit

Before the upgrade begins, each branch must be audited. The purpose of the audit is to confirm that the existing hardware, software, storage, network and services are suitable for clustering.

The audit should record:

| Area             | Details to Confirm                                                   |
| ---------------- | -------------------------------------------------------------------- |
| Server hardware  | Model, serial number, CPU, RAM, local disks, RAID, NICs              |
| Firmware         | BIOS, iDRAC/BMC, RAID controller and NIC firmware                    |
| Operating system | Windows Server version, licence and patch level                      |
| Hyper-V          | Existing virtual switches, virtual machines and storage paths        |
| Network          | VLANs, IP addressing, switch ports and trunking                      |
| Storage          | Existing capacity, RAID configuration and performance                |
| Power            | UPS, PDU and separate power rail availability                        |
| Management       | iDRAC access, monitoring and admin access                            |
| Backup           | Backup jobs, retention and restore status                            |
| AD/DNS/DHCP      | Domain membership, DNS records, DHCP scopes and time synchronisation |

**Implication:**
The audit may identify branches that are not ready for the standard upgrade. These sites may need remediation before the third host and shared storage are installed.

### 2. Validate the existing Hyper-V hosts

The existing two Dell PowerEdge R670 hosts must be validated before the new server is added.

Validation tasks include:

* Confirm both hosts are healthy.
* Confirm Dell support and warranty status.
* Confirm Windows Server 2022 Datacenter is correctly licensed.
* Confirm Hyper-V is installed and operating correctly.
* Confirm Failover Clustering is installed or available.
* Confirm CPU, memory and storage capacity.
* Confirm VLAN configuration.
* Confirm DNS and NTP are working.
* Confirm iDRAC/BMC remote management access.
* Confirm current backups are successful.
* Confirm current monitoring is active.

**Implication:**
Adding a new server to an unstable environment would increase risk. The current hosts must be stable first.

---

### 3. Upgrade and standardise the existing hosts

The existing hosts should be brought to a standard baseline before clustering.

Required tasks include:

* Update BIOS.
* Update iDRAC/BMC firmware.
* Update RAID controller firmware.
* Update NIC firmware.
* Apply Windows Server updates.
* Confirm Hyper-V configuration.
* Confirm Failover Clustering feature.
* Validate NIC Teaming or Switch Embedded Teaming.
* Confirm VLAN 2 host management.
* Confirm VLAN 9 storage traffic.
* Confirm VM network trunking.
* Confirm isolated iDRAC/BMC network.
* Confirm monitoring agents.
* Confirm backup agents.
* Confirm event logging.

**Implication:**
Standardisation reduces support issues during national rollout. It also ensures that all cluster nodes behave consistently.

### 4. Install the third Hyper-V server

A third Dell PowerEdge R670 server must be installed at each branch.

Installation tasks include:

1. Rack the new server.
2. Connect dual power supplies to separate power rails.
3. Connect iDRAC/BMC to the isolated management switch.
4. Connect host management interfaces to VLAN 2.
5. Connect storage interfaces to VLAN 9.
6. Connect VM network interfaces to the trunked VM network.
7. Configure BIOS settings for virtualisation.
8. Configure RAID 1 boot storage.
9. Install Windows Server 2022 Datacenter.
10. Install the Hyper-V role.
11. Install the Failover Clustering feature.
12. Apply Windows updates and firmware updates.
13. Configure host networking.
14. Join the server to `corp.electric-petrol.ie`.
15. Add the server to monitoring.
16. Confirm remote iDRAC access.
17. Confirm event logging.

**Implication:**
The third host increases resilience and capacity, but also increases power, cooling, rack, licensing and support requirements at each branch.

---

### 5. Implement clustered shared storage

The customer requires clustered shared storage. This means that the virtual machine files must be accessible by all three Hyper-V hosts.

Storage implementation tasks include:

* Configure the storage network on VLAN 9.
* Confirm storage switch connectivity.
* Configure the shared storage platform.
* Present shared storage to all three hosts.
* Configure Cluster Shared Volumes.
* Confirm all hosts can access the same storage.
* Configure permissions and access control.
* Configure storage monitoring.
* Test storage performance.
* Update backup jobs.
* Document the storage layout.

**Implication:**
Shared storage allows virtual machines to fail over between hosts. However, shared storage also becomes a critical dependency. If it is not designed with redundancy, it can become a single point of failure.

---

### 6. Configure the three-node Hyper-V Failover Cluster

After the third server and shared storage are ready, the Failover Cluster must be configured.

Cluster tasks include:

* Run Microsoft Cluster Validation Wizard.
* Add HV01, HV02 and HV03 to the cluster.
* Configure the cluster name and IP address.
* Configure cluster networks.
* Configure Cluster Shared Volumes.
* Configure Live Migration.
* Make critical virtual machines highly available.
* Set preferred owners for critical virtual machines.
* Configure failover priority.
* Configure cluster quorum.
* Configure Cluster-Aware Updating.
* Confirm DNS registration.
* Confirm Active Directory permissions.
* Confirm monitoring integration.

**Implication:**
The cluster must be sized using an N+1 model. This means the branch must be able to lose one server and still run the required workload on the two remaining servers.

---

### 7. Migrate virtual machines to shared storage

Existing virtual machines must be moved from host-local storage to shared cluster storage.

Migration tasks include:

* Confirm backups before migration.
* Identify critical virtual machines.
* Schedule a maintenance window.
* Move VM storage to Cluster Shared Volumes.
* Make each VM highly available.
* Test VM start-up.
* Test network connectivity.
* Test application functionality.
* Confirm backup after migration.
* Update documentation.

Critical virtual machines include:

| VM    | Purpose                            |
| ----- | ---------------------------------- |
| DC1   | Primary Active Directory and DNS   |
| DC2   | Secondary Active Directory and DNS |
| DHCP1 | DHCP services                      |
| MGMT1 | Monitoring and management          |
| FILE1 | Backup and file storage            |

**Implication:**
Migration may require planned downtime. A full backup and backout plan are required before workloads are moved.

---

### 8. Reengineer AAA

Authentication, Authorisation and Accounting must be moved into the Active Directory domain design.

AAA tasks include:

* Join all Hyper-V hosts to `corp.electric-petrol.ie`.
* Create AD security groups for administration.
* Remove unnecessary local administrator access.
* Configure role-based access.
* Configure separate administrator accounts.
* Configure Group Policy.
* Configure audit logging.
* Configure RADIUS/NPS for network device authentication where required.
* Document break-glass access.
* Test authorised and unauthorised access.

**Implication:**
The new AAA design improves security and accountability. It also reduces the risk caused by shared local administrator accounts across multiple sites.

---

### 9. Update backup and business continuity procedures

The backup and disaster recovery design must be updated for the new three-node cluster.

Backup and BCDR tasks include:

* Update backup jobs for shared storage.
* Confirm all critical VMs are protected.
* Confirm System State backups for domain controllers.
* Confirm DNS and DHCP backups.
* Confirm backup retention.
* Confirm offsite backup copy.
* Test VM restore.
* Test Active Directory recovery process.
* Update BCDR documentation.
* Confirm monitoring alerts for failed backups.

Recommended recovery priority:

| Priority | Service                           |
| -------- | --------------------------------- |
| 1        | Active Directory                  |
| 2        | DNS                               |
| 3        | DHCP                              |
| 4        | Hyper-V Management                |
| 5        | Monitoring Services               |
| 6        | Remaining Infrastructure Services |

**Implication:**
Backups must be tested. The branch should not be accepted as upgraded until a restore test has been completed.

| VLAN    | Purpose                          |
| ------- | -------------------------------- |

| VLAN 6  | Infrastructure Servers           |

The existing network configuration must support management, storage and virtual machine traffic separation. VLAN 9 should be dedicated to storage traffic.

Each existing host has dual 1100W hot-plug power supplies. These should be connected to separate power rails:

* PSU 1 to Power Rail A
* PSU 2 to Power Rail B

The UPS and PDU capacity must be checked because the branch will move from two hosts to three hosts.

### Storage upgrade

The existing local RAID 6 storage is suitable for local capacity, but the new requirement is clustered shared storage.

Required storage changes include:

* Configure shared storage accessible by all three hosts.
* Configure Cluster Shared Volumes.
* Move VM files from local storage to shared storage.
* Configure storage monitoring.
* Update backup jobs.
* Test failover access.

The existing local storage should not remain the primary location for highly available virtual machines.

### Justification

The third server should match the existing hosts because:

* It keeps the cluster hardware consistent.
* It reduces compatibility issues.
* It simplifies support.
* It simplifies firmware management.
* It supports predictable performance.
* It improves Live Migration compatibility.
* It keeps cabling and power standards consistent.
* It allows standardised documentation.

### Capacity after adding the third server

Each host provides 48 cores and 512GB RAM.

| Resource | Total Cluster Capacity |
| -------- | ---------------------- |
| CPU      | 144 cores              |
| RAM      | 1.5TB                  |

However, the branch must survive one host failure. The critical workload should therefore be able to run on two hosts:

| Resource | Available During One Host Failure |
| -------- | --------------------------------- |
| CPU      | 96 cores                          |
| RAM      | 1TB                               |

The cluster should not be designed to consume all available resources during normal operation. Spare capacity must be reserved for failover.

## A description of the new AD design.

The customer has specified that AAA must become part of the domain tree for:

electric-petrol.ie

The recommended internal Active Directory domain is:

corp.electric-petrol.ie

This internal domain sits under the wider Electric-Petrol.ie namespace.

electric-petrol.ie

&darr;

corp.electric-petrol.ie

This design separates internal infrastructure from public-facing services while keeping the company identity consistent.

### Domain design

The internal domain `corp.electric-petrol.ie` should support:

* Data centres
* Head office
* Branch sites
* Users
* Computers
* Servers
* Service accounts
* Administrator accounts
* Group Policy
* DNS
* AAA

Separate branch domains are not recommended because there are more than 60 service stations. Creating many domains would increase complexity and make management harder. A single internal corporate domain with a clear OU and site design is more suitable.

### Domain controller placement

Domain controllers should be placed in both data centres.

| Location     | Domain Controller Role                           |
| ------------ | ------------------------------------------------ |
| Mayo DC1     | Primary AD DS, DNS and authentication services   |
| Malin DC2    | Secondary AD DS, DNS and authentication services |
| Head Office  | Optional additional DC depending on user count   |
| Branch Sites | Optional RODC only where required                |

Mayo and Malin should both host domain controllers so authentication and DNS remain available if one data centre is unavailable.

For most small service stations, local writable domain controllers are not recommended. If a branch requires authentication during a WAN outage, a Read-Only Domain Controller may be used.

### Read-Only Domain Controllers

A Read-Only Domain Controller may be used at selected branch sites.

RODC benefits include:

* Local authentication during WAN issues.
* Lower security risk than a writable DC.
* Controlled password replication.
* Reduced exposure if a branch server is compromised.
* Improved logon performance at selected sites.

RODCs should only be used where there is a clear business requirement. Deploying RODCs to all branches may create unnecessary management overhead.

### DNS design

DNS should be AD-integrated and hosted on domain controllers.

Internal DNS zone:

corp.electric-petrol.ie

DNS requirements include:

* AD-integrated DNS.
* Secure dynamic updates.
* DNS hosted in Mayo and Malin.
* Forward lookup zone for `corp.electric-petrol.ie`.
* Reverse lookup zones for infrastructure networks.
* DNS records for hosts, clusters, storage and VMs.
* Branch clients configured to use internal DNS.

DNS is critical because Active Directory, Hyper-V, clustering and management systems depend on name resolution.

### DHCP design

DHCP should support the standard branch VLAN design.

DHCP should provide scopes for:

* VLAN 5 Technical Clients
* VLAN 6 Infrastructure Servers where required
* VLAN 12 Wireless AP Management
* Additional branch client VLANs where required

DHCP should include:

* Authorised DHCP servers.
* Exclusions for infrastructure devices.
* Reservations for known devices where needed.
* DNS server options.
* Default gateway options.
* Standard lease duration.
* DHCP failover where required.

### NTP design

Time synchronisation is critical for Active Directory authentication, Kerberos, logs and audit trails.

Recommended NTP design:

* Primary domain controller synchronises with a trusted external time source.
* Domain controllers provide time to domain members.
* Hyper-V hosts synchronise through the domain hierarchy.
* Infrastructure, storage and network devices use approved internal NTP sources.

### AD Sites and Services

AD Sites and Services should match the WAN topology.

Recommended AD sites:

* Malin-DC2
* HeadOffice
* Regional branch sites

For more than 60 service stations, it may not be practical to create an AD site for every small branch. A regional model may be easier to manage unless individual branch authentication requirements justify separate sites.

AD Sites and Services should control:

* Domain controller selection.
* Replication paths.

### AAA design

AAA stands for:

* Authentication
* Authorisation
* Accounting

The new AAA design should use Active Directory.

#### Authentication

Authentication should be handled through `corp.electric-petrol.ie`.

Authentication controls include:

* Domain-joined servers.
* Named user accounts.
* Separate administrator accounts.
* Strong password policy.
* MFA where supported.
* Restricted local logon.
* No shared administrator accounts.
* Break-glass accounts for emergencies only.

Example account model:

| Account Type        | Example           |
| ------------------- | ----------------- |
| Normal user account | liam.saunders     |
| Admin account       | adm-liam.saunders |
| Service account     | svc-backup-branch |
| Emergency account   | breakglass-branch |

#### Authorisation

Authorisation should be controlled through AD security groups.

Recommended groups:

| Group                      | Purpose                           |
| -------------------------- | --------------------------------- |
| GG-HyperV-Admins           | Hyper-V administration            |
| GG-Cluster-Admins          | Failover Cluster administration   |
| GG-Branch-Server-Admins    | Branch server administration      |
| GG-Network-Admins          | Switch and network administration |
| GG-Storage-Admins          | Shared storage administration     |
| GG-Backup-Operators        | Backup and restore                |
| GG-Monitoring-ReadOnly     | Read-only monitoring access       |
| GG-Helpdesk-Branch-Support | Limited branch support            |

Access should follow the principle of least privilege.

#### Accounting

Accounting should track administrator and user actions.

The design should log:

* Successful logons.
* Failed logons.
* Administrator logons.
* Changes to AD groups.
* GPO changes.
* VM power actions.
* VM migration actions.
* Cluster configuration changes.
* Storage configuration changes.
* Backup job activity.
* Network device logons.
* iDRAC/BMC access.
* DNS and DHCP changes.

Logs should be forwarded centrally where possible.

### RADIUS / NPS design

Network device administration should be integrated with Active Directory using RADIUS/NPS.

Recommended design:

* Deploy NPS servers in Mayo and Malin.
* Configure switches, firewalls and wireless controllers as RADIUS clients.
* Use AD groups to control access.
* Log administrator access.
* Keep local emergency accounts for break-glass use only.

This avoids shared local passwords on network devices and improves auditability.

### Branch authentication during WAN failure

Because major services are centralised, WAN failure must be considered.

| Option                          | Use Case                                  |
| ------------------------------- | ----------------------------------------- |
| Authenticate to data centre DCs | Normal operation                          |
| Cached credentials              | Basic workstation logon during WAN outage |
| RODC at selected branch         | Larger or critical branch                 |
| Local break-glass account       | Emergency infrastructure access           |

For most small service stations, cached credentials and reliable WAN may be enough. For larger or critical branches, RODC deployment can be considered.

## Conclusion

This design upgrades each Electric-Petrol.ie branch from a two-host Hyper-V environment to a three-node Hyper-V Failover Cluster with clustered shared storage. It meets the customer requirement that each site must continue operating after a single server failure.

The existing Dell PowerEdge R670 servers can be retained, but they must be upgraded, patched and standardised. The new third server should match the existing specification to maintain supportability and consistent performance.

The management implementation plan is provided as a separate XLSX file and should be used to control the upgrade from pilot branch through national rollout.

The Active Directory design uses `corp.electric-petrol.ie` as the internal domain under the wider `electric-petrol.ie` domain tree. This supports centralised authentication, authorisation and accounting across data centres, head office and branch sites.

A pilot branch should be completed first. After successful testing and sign-off, the upgrade should be rolled out nationally in controlled batches.
